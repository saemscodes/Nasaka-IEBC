The architecture is directionally sound but has specific gaps for your exact problem — markers displaced many km from actual locations. Here's the breakdown:

What holds up

Multi-source resolver with 5 sources is the right instinct — no single API has complete Kenya coverage
Confidence ≥ 0.7 threshold before auto-writing to Supabase is correct — prevents bad auto-corrections
HITL (Human-in-the-Loop) fallback is essential for a civic/electoral dataset
Audit log on apply is non-negotiable for IEBC data integrity
Re-verify loop after applying is good practice
What has problems

"Clustering < 200m" as an issue type — Your problem is markers displaced many km. Clustering < 200m is a duplicate detection problem, not a wrong-coordinate problem. These are different issue types requiring different resolution paths. The < 200m branch shouldn't feed the same resolver as a 50km displacement.

Gemini AI Geocoder as a source — Gemini doesn't have reliable grounded Kenya location data. It can hallucinate plausible-sounding but wrong coordinates, which then pollutes your consensus engine. This is the most dangerous node in the diagram for an electoral dataset.

Google Maps URL Parse — Valid only if your existing IEBC database records contain embedded Google Maps URLs. If the source of the URL is the same corrupted record, you're parsing bad data to fix bad data. Needs to be sourced externally (e.g., IEBC's own public portal URLs).

Consensus Engine is undefined — How is 0.7 confidence actually calculated? If 3/5 sources agree within 500m that's one thing. If 2 sources return coordinates and 3 return null, what's the score? This needs a concrete formula or the threshold is meaningless.

Null coords path — Null coordinates need a name-based lookup path (search by office name string), not the same coordinate-comparison resolver used for displaced markers. The architecture treats them the same.

HITL → "Admin Console Toggle" — Toggle what exactly? This is underspecified. The admin needs to: see the conflicting coordinates from each source, see the current bad entry, and either accept a suggested fix or manually enter coordinates. "Toggle" doesn't capture that.

Missing: displacement detection threshold — The entry point "Verification Detects Issue" needs to define how it detects a many-km displacement. The mechanism should be: geocode the office name → compare returned coords against stored coords → if delta > X km, flag. That delta threshold is not shown anywhere.

Recommended fixes

Split Issue Type into three distinct resolver paths: displaced coordinates (> threshold km), null coordinates (name lookup), and true duplicates (< 200m clustering). Remove Gemini as a geocoding source — replace with OpenCage or LocationIQ which have grounded OSM data. Define the consensus formula explicitly. Expand the HITL node into a proper admin review UI spec.

Architected comprehensive geocoding service implementation strategy
Architected comprehensive geocoding service implementation strategy
What's fully implemented:

5 geocoding sources — Nominatim (1 req/s, Kenya bounding box validation, User-Agent header per OSM policy), Google Maps URL (redirect + body regex parse fallback), Geocode.xyz (1 req/s, region=KE), Geokeo (daily counter with reset, apiKey from env), Gemini 1.5 Flash (temp=0, JSON-only prompt, null-safety parsing).

Rate limiters — Per-source RateLimiter class with per-second throttle and daily cap. Geokeo tracks 2,500/day with midnight reset. All are queue-based, not setTimeout spam.

Kenya bounding box guard — Every source validates lat: -5.0→5.0, lng: 33.9→42.0 before returning a result. Nothing outside Kenya passes through.

Consensus engine — Groups results within 1km radius, picks highest-aggregate-confidence group, computes weighted composite score: agreement(60%) + source_confidence(40%). Returns spread in km.

Issue type detection — Three distinct paths: NULL_COORDS (name-lookup path), DISPLACED (> 5km delta), CLUSTERING (< 200m to another office). Each routes differently, not conflated as the diagram had them.

Auto-apply vs HITL — Confidence ≥ 0.7 → writes to nasaka_iebc_offices, logs audit. Below threshold → writes to nasaka_hitl_queue with all source details for admin review.

Re-verify loop — After auto-apply, runs resolver again on the new coordinates and logs a _REVERIFY audit entry.

HITL management — fetchHITLQueue(), resolveHITL(), dismissHITL() all wired to Supabase. Admin can approve with custom coords, or dismiss with a reason.

Supabase tables needed: nasaka_iebc_offices, nasaka_geocode_audit, nasaka_hitl_queue.

Recognized misinterpretation; acknowledged preference for clarification before proceeding
Nominatim

One request per second hard limit — no exceptions, OSM will ban your IP. Always send a User-Agent header identifying your app (NasakaIEBC/1.0), it's required by their policy. Always append countrycodes=ke to every query to prevent returning results from other countries. After getting coordinates back, validate they fall within Kenya's bounding box (lat: -5.0 to 5.0, lng: 33.9 to 42.0) before trusting them — Nominatim sometimes returns results for similarly-named places in other countries. Build your query string as {office name} IEBC office {constituency} {county} county Kenya for best match specificity.

Google Maps URL Parse

No auth, no rate limit — but CORS will block this from the browser. This must run server-side (Supabase Edge Function or Node backend). The redirect URL from https://www.google.com/maps/search/{query} contains /@lat,lng in the final URL. Parse that with regex /@(-?\d+\.\d+),(-?\d+\.\d+). If the redirect doesn't expose coordinates in the URL, fall back to scanning the response body for the same pattern. This source is only as good as your query string — same Kenya bounding box validation applies.

Geocode.xyz

One request per second. Append &region=KE to bias results toward Kenya. The response returns latt and longt (note the extra t) — not lat/lng. It also returns a confidence field as a 0–100 integer; normalize it to 0–1 before feeding into your consensus engine. If the response contains an error key, treat the entire result as null. Rural Kenya coverage is weaker than Nominatim here.

Geokeo

2,500 requests per day total — track this with a counter that resets at midnight. The API key goes in the query string as &api=YOUR_KEY. Response structure nests coordinates at results[0].geometry.location.lat and .lng. Check data.status === "200" (it's a string, not a number) before parsing. This source is best used as a tertiary fallback after Nominatim and Geocode.xyz, to preserve the daily quota.

Gemini AI

60 RPM limit. The critical handling concern here is hallucination — Gemini will confidently return plausible-sounding but completely fabricated coordinates. Mitigate this by: setting temperature: 0.0 in the API call to eliminate creativity, requiring it to return {"lat": null, "lng": null} explicitly when uncertain rather than guessing, and assigning it the lowest confidence weight in your consensus engine (0.4 vs Nominatim's higher score). Always apply the Kenya bounding box check — hallucinated coordinates frequently land outside Kenya entirely. Parse the response by extracting the JSON object from the text with regex first, then JSON.parse, because Gemini will sometimes wrap the JSON in prose regardless of instructions.

Across all five — the universal rules:

Wrap every source in an independent try/catch so one failure doesn't abort the others. Run all five with Promise.allSettled, never Promise.all. A source returning null is valid — it means no result, not a crash. Never trust a single source in isolation; the consensus of agreeing sources is the actual answer.

Crawl entire websites with a single API call using Browser Rendering
Mar 10, 2026
Edit: this post has been edited to clarify crawling behavior with respect to site guidance.
You can now crawl an entire website with a single API call using Browser Rendering's new /crawl endpoint, available in open beta. Submit a starting URL, and pages are automatically discovered, rendered in a headless browser, and returned in multiple formats, including HTML, Markdown, and structured JSON. The endpoint is a signed-agent ↗ that respects robots.txt and AI Crawl Control ↗ by default, making it easy for developers to comply with website rules, and making it less likely for crawlers to ignore web-owner guidance. This is great for training models, building RAG pipelines, and researching or monitoring content across a site.
Crawl jobs run asynchronously. You submit a URL, receive a job ID, and check back for results as pages are processed.
Terminal window

# Initiate a crawl
curl -X POST 'https://api.cloudflare.com/client/v4/accounts/{account_id}/browser-rendering/crawl' \
  -H 'Authorization: Bearer <apiToken>' \
  -H 'Content-Type: application/json' \
  -d '{
    "url": "https://blog.cloudflare.com/"
  }'
 
# Check results
curl -X GET 'https://api.cloudflare.com/client/v4/accounts/{account_id}/browser-rendering/crawl/{job_id}' \
  -H 'Authorization: Bearer <apiToken>'
Key features:
* Multiple output formats - Return crawled content as HTML, Markdown, and structured JSON (powered by Workers AI)
* Crawl scope controls - Configure crawl depth, page limits, and wildcard patterns to include or exclude specific URL paths
* Automatic page discovery - Discovers URLs from sitemaps, page links, or both
* Incremental crawling - Use modifiedSince and maxAge to skip pages that haven't changed or were recently fetched, saving time and cost on repeated crawls
* Static mode - Set render: false to fetch static HTML without spinning up a browser, for faster crawling of static sites
* Well-behaved bot - Honors robots.txt directives, including crawl-delay
Available on both the Workers Free and Paid plans.
Note: the /crawl endpoint cannot bypass Cloudflare bot detection or captchas, and self-identifies as a bot.
To get started, refer to the crawl endpoint documentation. If you are setting up your own site to be crawled, review the robots.txt and sitemaps best practices.

Point-in-polygon for constituencies requires either: importing Kenya constituency GeoJSON into a Supabase table and using PostGIS ST_Contains, or calling Nominatim's reverse geocoding and parsing the address.county/address.state_district fields, or maintaining a flat lookup table of constituency centroids and returning the nearest one as an approximation. The /coordinates GeoJSON/CSV Format

The format switch (?format=geojson, ?format=csv) needs to be handled in the handler itself — Supabase returns JSON rows, you transform them. Why it matters for your context specifically

Most of your Kenyan institutional clients — county governments, NGOs, research orgs — will not have corporate cards for Stripe. M-Pesa is the realistic payment method for the Researcher ($49/mo = ~KES 6,300) and Institutional ($199/mo = ~KES 25,700) tiers. Daraja API prerequisites

You need a Safaricom Daraja account at developer.safaricom.co.ke. Create an app there — you get a Consumer Key and Consumer Secret. For STK Push specifically you need the Lipa na M-Pesa Online (MPESA Express) API. You also need:

A Paybill or Till Number — you must apply for this separately through Safaricom Business. This is not instant. Budget 2–4 weeks for approval. You need a registered business for Paybill. Till Number is faster but harder to reconcile programmatically.
A Passkey — Safaricom issues this specifically for your Paybill/Till after approval.
A publicly accessible callback URL — Safaricom's servers must be able to reach it. Localhost won't work. Your Vercel deployment URL works fine.
Sandbox credentials are available immediately for testing without a real Paybill.

How STK Push actually works

Your backend calls Daraja's /mpesa/stkpush/v1/processrequest with the amount, phone number, and your Paybill details
Safaricom sends an STK prompt to the user's phone
User enters their M-Pesa PIN
Safaricom hits your callback URL with success or failure
Your backend processes the callback and activates the subscription
The critical thing: the flow is asynchronous and callback-driven. You cannot confirm payment in the same HTTP request that initiates STK. Your frontend must poll or use websockets/realtime to know when payment is confirmed.

New environment variables needed

MPESA_CONSUMER_KEY, MPESA_CONSUMER_SECRET, MPESA_PASSKEY, MPESA_SHORTCODE (your Paybill/Till), MPESA_CALLBACK_URL (your Vercel endpoint), MPESA_ENVIRONMENT (sandbox or production).

New database tables needed

nasaka_mpesa_transactions: id, checkout_request_id (from Daraja response, your correlation key), merchant_request_id, phone_number, amount, api_key_id, tier_requested, status (pending/success/failed/cancelled), mpesa_receipt_number (null until callback), result_code, result_desc, initiated_at, completed_at.

New API endpoints needed

/api/mpesa/initiate — accepts phone, tier, api_key_id, calls Daraja, returns checkout_request_id to frontend for polling.

/api/mpesa/callback — Safaricom POSTs here. Verifies the payload, updates nasaka_mpesa_transactions, then if successful updates nasaka_api_keys tier and activates the subscription. Must return {"ResultCode": 0, "ResultDesc": "Accepted"} — if it doesn't, Safaricom retries.

/api/mpesa/status/:checkoutRequestId — frontend polls this to know if payment completed. Returns pending, success, or failed.

The subscription problem

M-Pesa has no native recurring billing. Stripe handles subscriptions automatically — M-Pesa does not. You have two options:

Manual renewal — user gets an email/SMS 3 days before expiry prompting them to pay again. Simple to build, high churn risk.

Standing Order / Direct Debit via M-Pesa — Safaricom has a Standing Order API but it's restricted, requires additional Daraja approval, and is not available to all businesses. Don't count on it.

Realistic approach for your scale: implement monthly manual renewal with automated reminders via email (you already have email templates from NOD). Lock the API key 3 days after expiry with a grace period, not immediately. This is the standard approach for Kenyan SaaS at your stage.

Phone number formatting

Daraja requires numbers in 2547XXXXXXXX format — no +, no leading 0. You must normalize on your backend: strip +, replace leading 0 with 254. A user entering 0712345678 must become 254712345678 before the Daraja call.

Amount handling

Daraja only accepts whole integer amounts in KES. Your tiers are $49 and $199. You need a KES equivalent that you fix or dynamically convert. Dynamic conversion (hitting an FX API on every payment) introduces risk — if the rate moves mid-checkout the amount the user sees and the amount charged can differ. Safer to fix KES amounts periodically (monthly) and display them explicitly: "KES 6,500/mo" rather than "$49/mo" for M-Pesa payers. Maintain two price displays on PricingPage.tsx.

Callback security

Safaricom does not sign their callbacks with a secret the way Stripe does. Anyone who knows your callback URL can POST fake success payloads. Mitigate this by: validating that checkout_request_id in the callback exists in your nasaka_mpesa_transactions table and is in pending status, and optionally calling Daraja's query API to independently confirm the transaction before activating the key.

Paystack webhook security

Unlike Stripe's header signature, Paystack uses a simpler model: they send an x-paystack-signature header which is a HMAC SHA512 hash of the raw request body using your secret key. Verify it like this on every webhook:

typescript
import crypto from 'crypto';

const hash = crypto
  .createHmac('sha512', process.env.PAYSTACK_SECRET_KEY!)
  .update(rawBody)
  .digest('hex');

if (hash !== req.headers['x-paystack-signature']) {
  return new Response('Unauthorized', { status: 401 });
}
You must use the raw request body — not JSON.parsed. In Vercel Edge Functions, read it as await req.text() before parsing.

**What you need to do before opening paid tiers**

Upgrade Supabase to Pro and switch your connection string to use Supavisor pooling (Supabase gives you a separate pooler connection string — use that one in your Vercel env vars, not the direct connection). Add `Cache-Control` headers to `/stats`, `/counties`, `/offices`, `/coordinates`. Implement Upstash Redis on the paid tier. That three-change sequence takes you from "breaks under one Taifa client" to "handles your full pricing tier stack."
